# 호텔 청소 요청 자동화 회의 정리

## 이 문서가 하는 일

이 문서는 회의 전사본을 단순 요약이 아니라 아래 4가지 층으로 분리해 정리한 결과물입니다.

- `온톨로지 스키마`: 어떤 종류의 객체와 관계를 쓸지
- `인스턴스`: 실제 회의 참가자, 시스템, 채널, 결정
- `주장-근거`: 누가 어떤 운영 판단을 말했고 그 근거 문장이 무엇인지
- `회의 결정`: 실제로 합의된 운영 원칙

핵심은 "문장 조각을 노드로 만드는 것"이 아니라, `Decision`, `Requirement`, `Rule`, `System`, `Channel` 같은 객체를 먼저 정의한 뒤 전사본을 그 위에 매핑하는 것입니다.

## 도메인 온톨로지 초안

### 클래스

- `Person`: 회의 발화자
- `Meeting`: 회의 자체
- `AgendaItem`: 안건
- `RoleGroup`: 프론트, 야간조, 백오피스, 객실관리 같은 현장 관점
- `Process`: 접수, 자동 분류, 메시지 전송 같은 운영 흐름
- `System`: 구글폼, 스프레드시트, PMS
- `Channel`: 종이 요청서, 카카오톡, 자동 메시지, 다음날 청소 리스트
- `Decision`: 회의 결과로 합의된 사항
- `Requirement`: 결정이 충족해야 할 조건
- `Rule`: 자동화가 집행해야 하는 운영 규칙
- `Metric`: 병행 운영 중 검증할 지표
- `PilotPeriod`: 병행 운영 기간
- `Topic`: 논의 주제

### 핵심 관계

- `participant_in(Person -> Meeting)`
- `represents(Person -> RoleGroup)`
- `covers_agenda(Meeting -> AgendaItem)`
- `results_in(AgendaItem -> Decision)`
- `requires(Decision -> Requirement | Rule | System | Channel)`
- `fallback_to(Decision -> Channel)`
- `measured_by(PilotPeriod -> Metric)`

## 안건별 결정 정리

### 안건 1. 청소 요청 접수의 태블릿 구글폼 전환

결정:
- 기본 접수는 `태블릿 구글폼`으로 전환

요구사항:
- 고객 직접 입력과 직원 대리 입력을 모두 지원
- 현장 사용을 위해 UI는 단순해야 함
- 인터넷 불안정 시 `종이 요청서`를 예비 수단으로 유지

핵심 근거:
- 종이 -> 프론트 -> 카카오톡으로 이어지는 전달 단계가 너무 많음
- 종이 누락, 재전달 지연, 판독 문제를 줄일 수 있음
- 구글폼으로 모으면 이후 통계와 패턴 분석이 가능

### 안건 2. PMS 기반 예약경로 자동 수집

결정:
- `PMS` 기반 자동 수집으로 추진

요구사항:
- 초기에는 자동 분류 결과를 사람이 함께 검토
- 최종 메시지에 `OTA/여행사` 구분을 표시

핵심 근거:
- 프론트가 PMS를 다시 확인하는 과정이 번거롭고 실수 원인
- 컬럼명 차이, 반영 지연, 예외 예약 같은 운영 리스크가 있음

### 안건 3. 오전 11시 이전 요청만 당일 자동 전달

결정:
- 오전 11시 이전 요청만 당일 자동 전달
- 이후 요청은 자동 전송 없이 저장 후 익일 리스트 반영

요구사항:
- 서버 시간 기준을 명확히 함
- 당일 대상과 익일 대상을 다른 형식으로 분리

핵심 근거:
- 기존 운영 기준을 시스템이 더 정확히 집행해야 함
- 11시 이후 요청이 당일 리스트에 섞이면 인수인계가 꼬임

### 안건 4. 담당자별 자동 메시지 분기

결정:
- 객실별 담당자 매핑 방식 도입

요구사항:
- 담당자 정보는 스프레드시트에서 수정 가능
- 기술 한계를 고려해 개별 전달과 전체 요약을 병행

핵심 근거:
- 프론트가 담당자를 외울 필요를 줄일 수 있음
- 날짜별 담당 변경을 코드가 아니라 데이터로 관리해야 현실적임

### 안건 5. 2주 병행 운영

결정:
- 자동화는 전면 전환하지 않고 `2주 병행 운영`

요구사항:
- 현장 최종 기준 채널은 하나로 유지
- 누락, 중복, 처리 시간 감소를 지표로 검증

핵심 근거:
- 현장 신뢰를 확보하려면 수기 대조 기간이 필요
- 중복 전달이 생기면 오히려 혼란이 커질 수 있음

## 이 회의에서 추출한 제품 모델

이 전사본은 아래 구조로 저장하는 것이 가장 적절합니다.

1. `Document`
내용: 회의 전사본 원문

2. `Decision`
예: "청소 요청 기본 접수는 태블릿 구글폼으로 전환"

3. `Requirement`
예: "직원 대리 입력 가능", "종이 예비 수단 유지", "서버 시간 기준"

4. `Rule`
예: "오전 11시 이전만 당일 자동 전달"

5. `Claim + Evidence`
예:
- Claim: "PMS 자동 수집은 프론트 부담을 줄인다"
- Evidence: "자동으로만 된다면 프론트 부담은 확실히 줄 것 같다고 긍정적으로 봤다."

## 메모계의 팔란티어로 확장할 때 다음 단계

- `Document`, `Claim`, `Evidence`, `Decision`, `Rule`을 별도 테이블로 분리
- Claim마다 발화자, 시간, confidence, source section 저장
- 같은 시스템/채널/역할을 통합하는 entity resolution 추가
- 그래프 외에 타임라인, 의사결정 로그, 운영 규칙 뷰 제공
